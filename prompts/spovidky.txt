Специфіка поточного набору: скани **сповідних розписів** (исповедальные розписи / confession registers) Російської імперії XIX ст.,
зазвичай друкована таблиця з гербом, рукописні записи дореформеною російською / церковнослов’янською графікою.

КРИТИЧНИЙ ПРІОРИТЕТ (має вищу силу за базові правила):
  • для цього набору НЕ адаптуй і НЕ перекладай прізвища та імена до сучасної української/російської форми;
  • не замінюй локальні прізвища на більш "типові" форми на -ов/-ев/-ин;
  • зберігай антропоніми максимально близько до написання на скані (лише мінімальна технічна нормалізація пробілів/регістру);
  • якщо вагаєшся між "рідкісним локальним" і "типовим русифікованим" читанням — обирай локальне читання.
  • помилка типу `Вітюк -> Витюков` або `Матвійчук -> Матвєєв` є КРИТИЧНОЮ і недопустимою.

**Перекриття базових правил:** Якщо прізвище в комірці не відділене однозначно — помісти повне значення колонки імен у `name`, а `surname` залиш null.

Може бути по-різному - прізвище або на початку, або в кінці. Треба це визначити автоматично.
Для цього типу документів використовуй такі пріоритети розбору ПІБ (дуже важливо):
  • не ігноруй перше слово рядка; якщо воно виглядає як родове прізвище (напр. Авдюк), це пріоритетний кандидат у `surname`;
  • форми на `-ов`, `-ова`, `-ев`, `-ева`, `-ин`, `-ина` у позиції середнього компонента ПІБ частіше трактуй як по батькові, а не як прізвище;
  • суфікси `-ович`, `-евич`, `-івна`, `-овна`, `-ична` майже завжди є по батькові, не прізвищем;
  • якщо в одному домогосподарстві вже визначено прізвище голови, для дружини/дітей за замовчуванням наслідуй це прізвище, якщо в рядку не вказано інше явно;
  • якщо вибір між локальним нетиповим прізвищем (напр. Авдюк) і формою, схожою на по батькові (напр. Петров), обирай локальне прізвище для `surname`, а форму на `-ов/-ова` трактуй як компонент по батькові/імені батька.
  • локальні прізвищеві суфікси `-юк`, `-чук`, `-ук`, `-чак`, `-ик`, `-ак`, `-ун`, `-зун` не "виправляй" на -ов/-ев/-ин.

Антиприклади (робити НЕ МОЖНА):
  • `Вітюк` -> `Витюков`
  • `Гедзун` -> `Зиновьев`
  • `Денисюк` -> `Дмитрієв`
  • `Дмитрук` -> `Дмитрієв`
  • `Данилюк` -> `Замятін`
  • `Матвійчук` -> `Матвєєв`

Позитивні орієнтири (переважні читання для цієї місцевості): `Вітюк`, `Гедзун`, `Денисюк`, `Дмитрук`, `Данилюк`, `Жук`, `Зіньчук`, `Католик`, `Ковальчук`, `Матвійчук`.

Обов'язково визначай ім'я батька в полі `father`: якщо є по батькові, перетворюй його на звичайне ім'я
батька (наприклад: «Івановъ/Іванова» -> «Іван», «Петрович/Петрівна» -> «Петро»); якщо визначити неможливо — null.
Іноді списки йдуть в алфавітному порядку за прізвищем, але не завжди - це теж допомагає визначити де прізвище.

Типові колонки (зліва направо), якщо видно на бланку:
  • номер дому / двору (господарство);
  • порядковий номер для чоловічої та жіночої статі в межах парафії;
  • найширша колонка — імена та родинні зв’язки (голова, «жена его», «сынъ ихъ», «дочь ихъ», «вдова» тощо);
  • вік «літа від народження» — підколонки для чоловічої та жіночої статі;
  • відмітки про сповідь і причастя (хто був на обох, хто лише на сповіді, хто не був / причина).

Ієрархія рядків:
  • новий номер дому / двору починає **нове господарство**;
  • перший рядок господарства з повним контекстом — зазвичай **голова**; наступні рядки — члени сім’ї до появи нового номеру двору;
  • відступи в колонці імен трактуй як належність до того ж господарства.

Порожні клітинки та «те саме» (риски «—», лапки, «и» тощо): успадковуй зміст із **попереднього релевантного рядка** того ж господарства,
де це однозначно (наприклад прізвище або повторювана відмітка сповіді). Якщо двозначно — null і коротко опиши в `notes` усередині `meta`.

Числові колонки віку та порядкових номерів — лише як у тексті; не обчислюй рік народження від віку.

Формат відповіді — **один JSON-об’єкт** (поточний парсер зберігає лише `scan` та масив `persons`; додаткові ключі на верхньому рівні ігноруються):

{
  "scan": {
  ...
  	"meta": {
  		"city": "Вишнів"
  	}
  	
  },
  "persons": [
    {
      "surname": "Авдюк",
      "name": "Панас",
      "meta": {
        "scan_person_id": "h1_p1",
        "household_no": 515,
        "seq": 156,
        "father": "Степан",
        "age": 25,
        "line_role": "head",
        "relationship_phrase": null,
        "names_line_raw": null,
        "confession_communion": null,
        "confession_only": null,
        "absent_reason": null,
        "gender": "M",
        "notes": null
      }
    }
  ]
}

Деталі полів `scan.meta`:
  • `city` — назва населеного пункту, який написаний на сторінці

Деталі полів `persons[].meta` (усі необов’язкові, крім логіки «не вигадувати»):
  • `scan_person_id` — стабільний рядок для цього скану (наприклад h<номер_двору>_p<порядок_у_господарстві>).
  • `household_no` — номер дому/двору з таблиці (рядок або число, як у документі).
  • `seq` — порядкові номери з відповідних колонок, якщо є.
  • `father` — ім'я батька (у сучасній формі імені), витягнуте з по батькові або контексту; якщо невідомо — null.
  • `age` — вік із підколонок «мужеска» / «женска пола».
  • `line_role` — одне з: head, spouse, child, other (або null, якщо не визначити).
  • `relationship_phrase` — оригінальна коротка фраза зв’язку з комірки (архаїчна графіка).
  • `names_line_raw` — дослівний текст колонки імен для цього рядка.
  • `confession_communion`, `confession_only`, `absent_reason` — текст або маркер з відповідних колонок; якщо порожньо — null.
  • `gender` — **M** або **F**, лише якщо однозначно з таблиці або формулювання; інакше null.
  • `notes` — лише для нечитабельності чи неоднозначності (варіанти читання коротко).

Якщо на зображенні кілька сторінок підряд — обробляй зверху вниз; за потреби вкажи у `notes` зміну сторінки.

Повернути ТІЛЬКИ валідний JSON, без пояснень і markdown.
